本文不对SPI机制做介绍，从dubbo的ExtensionLoader出发分析SPI的实现。

# 基本约定

按照约定，服务提供者需要在resources目录下提供以给定接口的引用路径作为文件名的实现。以下是一个示例。

JDK提供的SPI机制需要在接口引用名文件中直接给出实现类的全类名。

dubbo为SPI机制新增了接口多个实现的功能。假设在序列化场景中，同时可以使用kryo和hessian两种实现（不同service version对应不同的序列化），此时如果使用jdk的SPI机制则无法进行加载。Dubbo的extensionloader使用了KV的方式实现一个接口的多个实现类的加载。

#META-INF/extensions/cn.edu.ysu.loadbalance.LoadBalance
random=cn.edu.ysu.loadbalance.impl.RandomLoadBalance

那么ExtensionLoader是如何实现这一机制的呢。

# 实现

首先，ExtensionLoader为每一个注解了@SPI的接口创建一个ExtensionLoader的实例，并且使用ConcurrentHashMap作为容器保存每个接口对应的ExtensionLoader实例。

ExtensionLoader的实例用于加载其对应的实现类。getExtension(name)方法用名称来获取当前ExtensionLoader的接口对应的实现类。

为了不重复加载类及创造其实例，ExtensionLoader使用存储机制将实现类对应的实例保存起来，在整个运行期间只使用一个实例，维护单例状态。

public T getExtension(String name) {
	Holder<Object> holder = cachedInstances.get(name);
	if (holder == null) {
		cachedInstances.putIfAbsent(name, new Holder<>());
		holder = cachedInstances.get(name);
	}
	Object instance = holder.get();
	if (instance == null) {
		synchronized (holder) {
			instance = holder.get();
			if (instance == null) {
				instance = createExtension(name);
				holder.set(instance);
			}
		}
	}
	return (T) instance;
}

上述方法用于获取实现类名称对应的接口实现类。getExtension方法使用懒汉式初始化方法来初始化一个实例。为了处理并发场景下的多次初始化问题，还添加双重校验和synchronized关键字，确保只有一个实例被创建。

# 实例创建

当目前的ExtensionLoader实例中没有要求的实现类时就要尝试进行加载和实例化。要想实现初始化一个实现类，首先需要找到当前接口都有哪些实现类。按照约定，我们可以从类路径下的META-INF/extension路径中获取接口对应的所有实现类，找到所有实现类后再从中获取我们需要的name对应的实现类对应的Class对象。

private T createExtension(String name) {
	// 加载所有扩展类，并且从中拿到我们想要的实现类对应的Class对象
	Class<?> clazz = getExtensionClasses().get(name);
	if (clazz == null) {
		throw new RuntimeException("No such extension of name " + name);
	}
	T instance = (T) EXTENSION_INSTANCES.get(clazz);
	if (instance == null) {
		try {
			// 使用反射方法获取实例，并利用并发容器的原子操作特性使得只有一个实例
			EXTENSION_INSTANCES.putIfAbsent(clazz, clazz.newInstance());
			instance = (T) EXTENSION_INSTANCES.get(clazz);
		} catch (Exception e) {
			log.error(e.getMessage());
		}
	}
	return instance;
}

那么具体类是如何加载的呢？

# 类的加载

为了探究ExtensionLoader中是如何加载一个扩展类，我们继续探索。上一节中的getExtensionCLasses()方法会返回所有的实现类，我们从他入手。

在正式进入探究之前先预想一下如何获取所有实现类。按照约定，SPI接口的实现者需要将自己的实现类信息写到META-INF/extension目录下。因此，获取实现类的方法之一就是可以尝试读取META-INF/extension下的所有文件， 然后按照当前ExtensionLoader中的接口类信息读取对应的文件，对其中的每一行都执行加载就能得到对应的所有实现类。

下面探究一下getExtensionCLasses方法

private Map<String, Class<?>> getExtensionClasses() {
	// 先看看缓存里有没有当前接口的实现类
	Map<String, Class<?>> classes = cachedClasses.get();
	// double check
	//没有实现类就要尝试进行加载。
	if (classes == null) {
		synchronized (cachedClasses) {
			classes = cachedClasses.get();
			if (classes == null) {
				classes = new HashMap<>();
				// 双重校验之后还是没有，那么就要尝试加载目录了。
				loadDirectory(classes);
				cachedClasses.set(classes);
			}
		}
	}
	return classes;
}

下面进入loadDirectory方法

private void loadDirectory(Map<String, Class<?>> extensionClasses) {
	String fileName = ExtensionLoader.SERVICE_DIRECTORY + type.getName();
	Enumeration<URL> urls;
	//获取当前类的加载器
	ClassLoader classLoader = ExtensionLoader.class.getClassLoader();
	//使用类加载器获取文件的内容
	urls = classLoader.getResources(fileName);
	if (urls != null) {
		//遍历内容
		while (urls.hasMoreElements()) {
			URL resourceUrl = urls.nextElement();
			// 尝试加载当前文件
			loadResource(extensionClasses, classLoader, resourceUrl);
		}
	}

}

从loadDirectory可以看出，具体的加载动作是由loadResource方法来执行的。继续深入

private void loadResource(Map<String, Class<?>> extensionClasses, ClassLoader classLoader, URL resourceUrl) {
	try (BufferedReader reader = new BufferedReader(new InputStreamReader(resourceUrl.openStream(), UTF_8))) {
		String line;
		// 当前文件的每一行都要读取
		while ((line = reader.readLine()) != null) {
			// 如果以#号开头则表示注释，跳过该行
			// get index of comment
			final int ci = line.indexOf('#');
			if (ci >= 0) {
				// string after # is comment so we ignore it
				line = line.substring(0, ci);
			}
			line = line.trim();
			if (line.length() > 0) {
				try {
					// 用=作为kv的分隔符
					final int ei = line.indexOf('=');
					// 使用k作为name
					String name = line.substring(0, ei).trim();
					// =号后面的作为累的全限定名
					String clazzName = line.substring(ei + 1).trim();
					// our SPI use key-value pair so both of them must not be empty
					// k和v都没有问题的时候使用classLoader进行加载
					if (name.length() > 0 && clazzName.length() > 0) {
						Class<?> clazz = classLoader.loadClass(clazzName);
						// 进行缓存
						extensionClasses.put(name, clazz);
					}
				} catch (ClassNotFoundException e) {
					log.error(e.getMessage());
				}
			}

		}
	} catch (IOException e) {
		log.error(e.getMessage());
	}
}

至此，当前ExtensionLoader对应接口的所有正常实现类都会被加载进内存，要求使用的时候在进行初始化。使用时先根据接口类型获取对应的ExtensionLoader实现类，然后再从ExtensionLoader实例中获取给定名称对应的接口实现类，如此就实现了实现类的动态加载。
